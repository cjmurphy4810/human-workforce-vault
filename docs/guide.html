Section 1

Overview & Purpose

The Problem This Solves

Podcast channels accumulate enormous bodies of knowledge that are nearly inaccessible in raw form. A subscriber who discovers @TheHumanWorkforce today faces 186 videos with no structured way to navigate themes, find episodes on a specific topic, identify the most important insights across the full corpus, or connect ideas across episodes recorded months apart.

The Human Workforce Knowledge Vault solves this by treating the full channel as a structured knowledge base — collecting, cleaning, analysing, connecting, and surfacing its intelligence through a set of automated, reproducible pipeline scripts powered by AI.

What Makes This Different From a Simple Transcript Dump

Each layer adds structured intelligence: transcripts → summaries → entities → graph → indexes → report. By the final step, Claude Sonnet is reading a compressed, curated view of 120,000+ characters of verified source material — not raw transcripts — which is why the intelligence report reads at a level of synthesis and specificity that a simple "summarise this video" prompt cannot achieve.

Section 3

Data Collection Layer

Script 01 01_collect_metadata.py — YouTube Metadata Collection
InputChannel URL in config.json
Outputyoutube/metadata/*.md + index.json
AI callsNone
API keyNot required

How it works: Uses yt-dlp --dump-json --flat-playlist to extract the full channel playlist without downloading any video files. For each video, it writes a Markdown metadata file (title, upload date, duration, URL, description) and updates a central index.json with a stable video_id key.

Why yt-dlp instead of the YouTube Data API: The YouTube Data API requires an API key with a 10,000 unit/day quota — a full channel crawl can exhaust this in one run. yt-dlp has no quota and no key requirement.

Validation approach: The script checks that the playlist extraction returned results and that each entry has a non-empty id field. The index.json is always regenerated fresh from the yt-dlp output, so it always reflects the live channel state.

Slug generation: Each video gets a URL-safe slug from its title via python-slugify, with a 60-character truncation. The filename format is YYYY-MM-DD-slug.md derived from the upload date — this makes the metadata directory sortable by date and prevents filename collisions.

Script 02 02_collect_transcripts.py — Transcript Extraction
Inputindex.json (video IDs)
Outputyoutube/transcripts/*.txt
AI callsNone
Success rate~41% (76/186 videos have captions)

Dual-source architecture: The script uses two transcript sources in sequence, falling back automatically if the primary fails:

  1. Primary: YouTubeTranscriptApi().fetch(video_id, languages=["en", "en-US"]) — the official YouTube caption API, fastest and most accurate. Note the instance method pattern (YouTubeTranscriptApi()) required by v1.2.4+ of the library.
  2. Fallback: yt-dlp --write-auto-sub --skip-download — downloads the auto-generated VTT subtitle file, which is then parsed to extract plain text.

Why the low success rate (41%)? The other 59% are YouTube Shorts — short vertical videos (under 60 seconds) that YouTube does not generate captions for. Full podcast episodes all have captions. This is expected and correct behaviour.

Cleaning pipeline applied to every transcript:

  • Remove [Music], [Applause], and other sound tags
  • Strip HTML tags from VTT files
  • Deduplicate consecutive identical lines (common in auto-caption VTT files)
  • Break continuous text into ~500-character paragraphs at sentence boundaries
  • Strip leading/trailing whitespace and normalize multiple blank lines

Validation: Each saved transcript file is verified to be non-empty after cleaning. If both sources fail, the episode is marked as transcript_failed in index.json and skipped in all downstream scripts — it does not cause a pipeline crash.

Key API Compatibility Note

youtube-transcript-api v1.2.4 removed the class method YouTubeTranscriptApi.get_transcript(). All scripts use the instance method pattern YouTubeTranscriptApi().fetch(). The fetched snippet objects are FetchedTranscriptSnippet dataclasses, converted to dicts via [{"text": s.text} for s in fetched] before processing. This is the pattern that works as of v1.2.4.

Section 4

AI Analysis Layer

Script 03 03_summarize_episodes.py — Episode Summarisation
Modelclaude-haiku-4-5-20251001
InputCleaned transcripts
Outputyoutube/summaries/*.md
Cost~$0.002 per episode

Why Haiku (not Sonnet) here: At this stage we are doing 76 independent extraction tasks on individual transcripts. Haiku is 15× cheaper than Sonnet and sufficient for structured extraction from a single document. Sonnet is reserved for the final cross-corpus synthesis (Script 10).

The 7-section prompt structure and why each section was chosen:

SectionPurpose
Executive Summary2–3 sentence overview for navigation and search
Key Themes3–5 thematic tags — feeds the master index and knowledge graph
Key Findings3–5 concrete takeaways — the substantive intelligence
Actionable TakeawaysPractitioner-oriented actions — increases usefulness
Notable QuotesVerbatim content — anchors summaries to source material
Human Workforce RelevanceChannel-specific framing — ensures output is domain-relevant
Suggested Future ContentContent strategy value — bonus output for the channel owner

Validation:

  • Truncation guard: Transcripts longer than 100,000 characters are truncated before sending. This prevents hitting the API's context window limit and keeps costs predictable.
  • Section presence check: The script verifies the response contains at least the "Executive Summary" header before saving.
  • Incremental processing: Episodes with an existing summary_file in index.json are skipped unless --force is passed. A failed API call does not corrupt the index.
  • Structured output: The 7-section Markdown format is enforced by the prompt. Haiku follows this structure reliably across all 76 episodes.
Script 04 04_extract_entities_topics.py — Entity & Topic Extraction
Modelclaude-haiku-4-5-20251001
Outputtaxonomy/entities.json + topics.json
API calls1 per episode (combined)

Single-call efficiency: Rather than making two separate API calls per episode (one for entities, one for topics), the prompt requests both in a single JSON response. This halves API costs and latency for this step.

Entity categories extracted:

  • people — Named individuals mentioned
  • organizations — Companies, institutions, agencies
  • products — Named software products and services
  • platforms — AI/cloud platforms (AWS, Azure, etc.)
  • concepts — Coined or specialized terms (e.g., "ghost in the org chart")
  • technologies — Technical domains and approaches
  • methodologies — Frameworks and processes
  • frameworks — Named analytical or governance frameworks

JSON fence stripping: Claude often wraps JSON responses in Markdown code fences (```json ... ```). The script strips these before parsing with a regex: re.sub(r'^```json?\n?|```$', '', response.strip(), flags=re.MULTILINE). This prevents json.JSONDecodeError on otherwise valid responses.

Validation:

  • Incremental: episodes already in entities.json are skipped (keyed by video_id)
  • Schema check: the response must have at least entities and topics keys
  • Failed episodes are logged but do not abort the run — the rest of the batch continues

Knowledge Architecture Layer

Script 05 05_build_knowledge_graph.py — Knowledge Graph Construction
AI callsNone
Outputtaxonomy/knowledge-graph.json
Result1,903 nodes · 5,842 edges

Node types and ID scheme:

TypeID FormatExample
episodeepisode:{video_id}episode:dQw4w9WgXcQ
conceptconcept:{category}:{label}concept:people:satya-nadella
topictopic:{topic_name}topic:AI Governance

Edge types:

RelationshipConnectsMeaning
mentionsepisode → conceptThis episode mentions this entity
coversepisode → topicThis episode covers this primary topic
relatedepisode ↔ episodeTwo episodes share 2+ primary topics

Related edge threshold: The "2+ shared topics" threshold was chosen to avoid generating trivial connections (any two governance episodes would be connected by "AI") while still capturing meaningful thematic clustering. Lowering to 1 would generate too many weak edges; raising to 3 would miss most connections given the typical 3–5 topics per episode.

Validation: This is a deterministic, pure-Python computation. The same inputs always produce the same graph. The script prints node and edge counts on completion — these can be compared against previous runs to detect anomalies (e.g., a bulk deletion of taxonomy data).

Script 06 06_build_master_index.py — Master Topic Index

Groups all episodes by their primary topics (from topics.json), then ranks the top 50 concepts by raw frequency across the full corpus (from entities.json). Output is a single Markdown file at topic-index/master-index.md designed to give a human (or NotebookLM) a complete navigational map of the vault in one document.

No AI calls. No external dependencies beyond the existing taxonomy files. Deterministic.

Script 07 07_build_speaker_index.py — Speaker Cross-Reference

Reads the speakers list from config.json and searches each cleaned transcript for occurrences of each speaker's name. For each speaker, outputs a list of episodes where they appear, with timestamps not required — simple name-match detection is sufficient for cross-reference purposes.

Limitation: This is a name-string search, not speaker diarization. If a transcript refers to someone as "he" or "the guest" rather than by name, that appearance will not be detected. For the purposes of a content vault, this level of detection is appropriate and low-cost.

Outputshorts/episode-NNN-slug-shorts.md

For each episode with a summary, generates 10 short-form content ideas in a structured 4-element format:

  • Hook — First 3 seconds that stops the scroll
  • Key Message — The core insight in one sentence
  • Suggested Visual — What to show on screen
  • CTA — Call to action for the specific insight

This output is channel-strategy intelligence — it turns the knowledge vault into an active content production tool, not just an archive.

Script 09 09_prepare_notebooklm.py — NotebookLM Source Files

Produces a set of structured Markdown files specifically designed for import into Google NotebookLM as a knowledge base:

  • notebooklm/master-source.md — All summaries combined in one searchable document
  • notebooklm/episode-index.md — Full episode listing with dates and topics
  • notebooklm/glossary.md — Key terms and concepts extracted from entities
  • notebooklm/frameworks.md — Named frameworks and methodologies
  • notebooklm/topic-guides/*.md — 16 topic-specific files, one per primary category

The master-source.md file is also the primary input to Script 10 (the intelligence report). Its first 120,000 characters are passed to Claude Sonnet as the knowledge corpus.

Script 10 10_generate_report.py — Intelligence Report
Modelclaude-sonnet-4-6
Max tokens8,192
Input chars120,000 (master-source.md)
Cost~$0.40 per run

Why Claude Sonnet here (not Haiku): The intelligence report requires synthesis across the full corpus — identifying cross-episode patterns, generating novel concept names ("ghost in the org chart"), making multi-horizon predictions, and writing at publication quality. This is the hardest cognitive task in the pipeline. Sonnet's reasoning depth produces qualitatively better outputs than Haiku would at this synthesis level.

The analysis prompt structure: The prompt instructs Claude Sonnet to act as a "Senior Research Analyst" (a persona that primes professional, analytical writing) and produce eight specific sections:

  1. Executive Summary (4 paragraphs, no bullet points — narrative synthesis)
  2. Top 25 Themes (numbered, titled, with description)
  3. 10 Most Important Findings (evidence-grounded)
  4. 8 Emerging Trends (with episode citations)
  5. 5 Workforce Predictions + 5 AI Predictions + 5 Governance Predictions
  6. 5 Recommended Future Books (with thesis and audience)
  7. 5 Recommended Future Podcast Series (with concept and episode ideas)
  8. 5 Recommended Future Courses (with outcomes)

What makes the report reliable:

  • Source fidelity: The 120k-character input is the verified, human-cleaned summaries from Script 03 — not raw transcripts. Claude is synthesising pre-validated, structured intelligence.
  • Topic distribution grounding: The prompt also includes the topics.json frequency distribution so the report reflects actual content proportions, not sampling bias toward memorable episodes.
  • Specificity requirement: The prompt explicitly instructs "cite specific episodes and evidence" in findings and trends, preventing generic outputs.
  • 8,192 token limit: Set high enough to allow complete, long-form sections without truncation.

Flags:

  • --force — Overwrite existing report (default: skip if report already exists)
  • --dry-run — Show what would be generated without making an API call
Security: API Key Handling

The Anthropic API key is never stored in any file in this repository. It is always read from the ANTHROPIC_API_KEY environment variable. The script exits with a non-zero error code if this variable is not set. For GitHub Actions, the key is stored as a repository secret (Settings → Secrets and Variables → Actions) and injected via ${{ secrets.ANTHROPIC_API_KEY }}.

Script 11 11_generate_html.py — GitHub Pages HTML Generator

Reads live vault data and the intelligence report markdown, parses each section, and regenerates docs/index.html with updated stats and content. This is the script that closes the loop between the pipeline and the public-facing website.

What it reads:

  • youtube/metadata/index.json — for video/transcript/summary counts
  • taxonomy/knowledge-graph.json — for node and edge counts
  • taxonomy/topics.json — for topic count
  • exports/human-workforce-intelligence-report.md — for all content sections

Markdown parsing approach: Uses Python re (standard library only — no dependencies). Splits the report by ## headers to isolate sections, then applies pattern-specific regex to each section to extract structured data (themes as (title, description) tuples, findings as (title, body) pairs, etc.).

cwd-first config: Like all pipeline scripts, it detects the project root by checking if youtube/metadata/ exists in the current working directory. This enables running from any directory and enables test isolation with tmp_path.

Section 7

Reliability Design Principles

Several engineering patterns work together to make the pipeline consistently reliable across repeated runs, new data, and edge cases.

1. Incremental Processing Pattern

Every AI-calling script (03, 04, 08) skips items that already have a result in the output file, checking by video_id before making any API call. This means:

  • Interrupted runs can be resumed without re-processing completed items
  • Adding 10 new videos to a 186-video channel only costs API calls for the 10 new episodes
  • A script that crashes on item 50 of 76 restarts from item 51, not item 1

The --force flag overrides this behaviour when a fresh regeneration is needed.

2. Dual-Source Transcript Validation

Script 02 uses two independent sources for transcripts. If the primary (youtube-transcript-api) returns no results or raises an exception, the fallback (yt-dlp VTT) is attempted automatically. This dual-source approach means a YouTube API change, rate limit, or regional restriction on one source will not silently produce empty transcripts — it will try the other source and only fail if both sources fail.

3. cwd-First Config Pattern

Every script discovers the project root by checking whether youtube/metadata/ exists in the current working directory before falling back to the script's parent directory:

_cwd = Path.cwd()
ROOT = _cwd if (_cwd / "youtube" / "metadata").exists() \
       else Path(__file__).resolve().parent.parent

This enables test isolation: tests use pytest's tmp_path fixture with a fake directory structure, so no test ever touches the real vault data or makes a real API call.

4. JSON Fence Stripping

Language models frequently wrap JSON responses in Markdown code fences even when the prompt says not to. Every script that parses JSON from an API response strips these fences before parsing:

raw = raw.strip()
if raw.startswith("```"):
    raw = re.sub(r'^```json?\n?', '', raw)
    raw = re.sub(r'\n?```$', '', raw)

Without this, roughly 30% of responses would fail to parse even though the underlying JSON is valid.

5. Non-Zero Exit Codes for Missing Prerequisites

Scripts that require prerequisites (ANTHROPIC_API_KEY, a previous script's output) check these at startup and exit with a non-zero code and clear error message if they are missing. This makes pipeline failures visible immediately rather than silently producing empty outputs.

6. --dry-run and --force Flags

All scripts support --dry-run (show what would happen without side effects) and AI scripts support --force (overwrite existing outputs). These flags make the pipeline safe to test and predictable to operate.

7. Pipeline State Management

The orchestrator (run_pipeline.py) writes a pipeline-state.json checkpoint file after each successful script. The --resume flag reads this state and skips already-completed steps. A --from=N flag starts the pipeline from step N. The state is always reset (not appended) on non-resume runs to prevent stale checkpoint corruption.

Section 8

Quality Assurance & Testing

TDD Approach

Each of the 10 core scripts has a corresponding test file in tests/. Tests were written and run before the implementation was considered complete (Test-Driven Development). This approach was critical for catching the youtube-transcript-api v1.2.4 API breaking change before it caused silent failures in production.

Test Isolation

Tests use pytest's tmp_path fixture to create temporary directory structures that mimic the real vault layout. The cwd-first config pattern ensures scripts use these temporary directories instead of the real vault. No test ever touches the actual vault data, makes a real API call, or requires internet access.

Key Bug Caught by Tests

youtube-transcript-api v1.2.4 Breaking Change

The test suite caught that YouTubeTranscriptApi.get_transcript() (class method) had been removed in v1.2.4. The fix was to use YouTubeTranscriptApi().fetch() (instance method) and convert FetchedTranscriptSnippet dataclass objects to dicts. Without TDD, this would have silently produced zero transcripts in production.

Section 9

GitHub Pages & Refresh Automation

How the Site is Served

The repository is configured to serve GitHub Pages from the /docs folder on the master branch. Any push to master that changes files in docs/ triggers an automatic GitHub Pages rebuild, typically deploying within 2–3 minutes.

The live URL is: https://cjmurphy4810.github.io/human-workforce-vault/

The Refresh Data Button

The ↻ Refresh Data button in the site header links to the GitHub Actions workflow dispatch page. Clicking it opens GitHub, where you click "Run workflow" to trigger a full pipeline refresh. The workflow:

  1. Checks out the repository
  2. Installs Python dependencies
  3. Runs Scripts 01–11 in sequence (AI scripts skip already-processed items)
  4. Commits changed files (metadata, taxonomy, exports, docs)
  5. Pushes to master
  6. GitHub Pages redeploys automatically

Setting Up the ANTHROPIC_API_KEY Secret

For the GitHub Actions workflow to make Claude API calls, you must add the API key as a repository secret:

  1. Go to github.com/cjmurphy4810/human-workforce-vault
  2. Click Settings → Secrets and Variables → Actions
  3. Click New repository secret
  4. Name: ANTHROPIC_API_KEY
  5. Value: your Anthropic API key from console.anthropic.com
Automatic Weekly Refresh

The workflow is also scheduled to run every Monday at 6 AM UTC via the cron: '0 6 * * 1' trigger. This means the vault automatically refreshes when new episodes are published, without manual intervention. You can disable this by removing the schedule: block from .github/workflows/refresh-vault.yml.

  - Node types: episode:{video_id}, concept:{category}:{label}, topic:{name}
  - Edge types: mentions (episode→concept), covers (episode→topic),
    related (episode↔episode when 2+ shared topics)
  - Save to taxonomy/knowledge-graph.json

06_build_master_index.py
  - Group episodes by topic, rank top 50 concepts by frequency
  - Output: topic-index/master-index.md

07_build_speaker_index.py
  - Search transcripts for speaker names from config["speakers"]
  - Output: speaker-index/speaker-index.md

08_generate_shorts.py
  - Model: claude-haiku-4-5-20251001
  - 10 short-form content ideas per episode
  - 4-element format: Hook / Key Message / Suggested Visual / CTA
  - Output: shorts/episode-NNN-slug-shorts.md

09_prepare_notebooklm.py
  - Combine all summaries into notebooklm/master-source.md
  - Create episode-index.md, glossary.md, frameworks.md
  - Create 16 topic-guides/*.md files (one per primary category)

10_generate_report.py
  - Model: claude-sonnet-4-6 (most capable — this is the final synthesis)
  - max_tokens: 8192
  - Read master-source.md (first 120k chars) + topics.json distribution
  - Role: "Senior Research Analyst at [YOUR CHANNEL NAME]"
  - Produce: Executive Summary, Top 25 Themes, 10 Key Findings,
    8 Emerging Trends, 5×3 Predictions (workforce/AI/governance),
    5 Future Books, 5 Future Podcast Series, 5 Future Courses
  - Flags: --force (overwrite existing), --dry-run (skip API call)
  - Exit non-zero if ANTHROPIC_API_KEY missing OR master-source.md absent

11_generate_html.py
  - Read live stats from index.json + knowledge-graph.json + topics.json
  - Parse intelligence report markdown with regex (stdlib only)
  - Generate docs/index.html with embedded CSS, sticky nav, card layouts
  - Embed generated_date = datetime.now().strftime("%B %d, %Y")
  - --dry-run flag supported

run_pipeline.py
  - Run scripts 01-11 in sequence
  - pipeline-state.json for checkpoints; --resume skips completed; --from=N
  - Reset state["completed"] = [] on non-resume runs
